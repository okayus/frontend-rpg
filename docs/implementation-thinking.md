# グリッドマップ＋キャラ移動を実装するときの思考プロセス

「CSS Grid でタイルマップを表示し、キーボードでキャラクターを動かす」機能を作るとき、  
どういう順番で何を考え、どんな前提知識に基づいて判断したかを解説する。

---

## 全体の思考の流れ

```
1. まず「何が登場するか」を型で定義する         ← TypeScript の型設計
2. 実際のマップデータを作る                     ← データ定義
3. 画面にどう並べるかを決める                   ← CSS レイアウト設計
4. コンポーネントの入出力（props）を決める       ← React コンポーネント設計
5. コンポーネントの中身を実装する               ← JSX + CSS の実装
6. ユーザー操作（キー入力）と状態変化を繋げる   ← React Hooks で状態管理
7. 全部を組み合わせる                           ← App で結合
```

この順番は「下のレイヤーから上に積む」考え方。  
型 → データ → 見た目 → 操作 の順に組み立てると、各段階で「前の段階が完成している」状態になり、手戻りが少ない。

---

## Step 1: 型を定義する —「何が登場するか」を決める

### 考えたこと

> マップとプレイヤーがある。マップはタイルの集まりで、タイルには種類がある。  
> プレイヤーには位置と向きがある。  
> まずこれらの「意味」を型として定義しよう。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **TypeScript では型を先に定義すると、以降のコードで型チェックが効く** | 型を最初に作れば、データ定義やコンポーネント実装時にミスをコンパイラが検出してくれる。型が後回しだと「動くけど壊れやすいコード」が先にできてしまう |
| **`as const` で定数オブジェクトを作るとリテラル型になる** | `TileType.Grass` が `number` ではなく `0` という具体的な型になる。enum より軽量で Tree Shaking も効く |
| **Union 型で取りうる値を限定できる** | `Direction = 'up' \| 'down' \| 'left' \| 'right'` にすれば、typo を型エラーで検出できる。`string` にすると何でも入ってしまう |
| **`readonly` で変更を禁止できる** | マップデータやプレイヤー座標はイミュータブル（不変）に扱いたい。React の状態管理では「直接変更せず新しいオブジェクトを作る」のが原則なので、readonly で誤った変更を防ぐ |
| **`interface` はオブジェクトの形を定義する** | `MapData`, `Position` のようなデータの構造には interface が自然。文字列の Union 型には type を使う |

### 判断のポイント

- **タイルの種類をどう表現するか？** → 数値の定数オブジェクト (`as const`)。  
  文字列 (`'grass'`, `'wall'`) でもいいが、マップデータを2次元配列で書くときに数値のほうがコンパクト。  
  将来 JSON から読み込む場合も数値が扱いやすい。

- **通行判定をどこに書くか？** → 型定義と同じファイルに `isPassable()` 関数を置いた。  
  「タイルの通行可否」はタイルの性質なので、タイルの型定義に近い場所が自然。  
  複数の場所（移動判定、敵の移動AI、将来のイベント判定など）から呼ばれる共通ロジック。

```
src/types/map.ts を作成:
  - TileType（タイル種別の定数）
  - isPassable()（通行可否の判定）
  - MapData（マップ全体のデータ構造）
  - Position（座標）
  - Direction（移動方向）
```

## Step 2: マップデータを作る —「どんなステージか」を定義する

### 考えたこと

> 型ができた。次は実際のマップデータを作ろう。  
> 16×12 のグリッドで、森に囲まれた草原に道と池と石壁がある感じにしよう。  
> データは見た目でマップの形がわかるように書きたい。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **データとロジックを分離する（関心の分離）** | マップデータは `data/` ディレクトリに、描画ロジックは `features/` に置く。データだけ差し替えれば別のマップが作れる |
| **分割代入で変数を短くできる** | `const { Grass: G, Wall: W, ... } = T` で1文字変数に展開。タイル配列をアスキーアート的に視認できるようにする |
| **型アノテーションでデータの正しさを保証する** | `const grasslandMap: MapData = { ... }` と書けば、必須プロパティの漏れや型の不一致をコンパイラが検出する |

### 判断のポイント

- **配列を `[y][x]` の順にしたのはなぜ？** → コード上の見た目とマップの形が一致するから。  
  外側のループが行（上から下）、内側が列（左から右）。  
  コードを見るだけで「上が森の壁、右に池がある」とわかる。

- **width / height を明示したのはなぜ？** → CSS Grid の列数設定に数値が必要だから。  
  `tiles[0].length` でも取れるが、「このマップは 16×12 として設計した」という意図の明示にもなる。

```
src/data/maps/grassland.ts を作成:
  - grasslandMap（16×12 の2次元タイル配列 + メタデータ）
```

## Step 3: 画面にどう並べるかを決める — CSS レイアウト設計

### 考えたこと

> タイルを格子状に並べたい。行×列の2次元レイアウトには CSS Grid が最適。  
> タイルサイズは CSS 変数にして、画面幅に応じて変えたい。  
> 画面幅の判定は Container Query を使えば、コンポーネント単位で応答できる。  
> タイルの色は種類ごとに変えて、微妙なバリエーションを color-mix() で作ろう。  
> スタイルの優先度は @layer で管理すれば、意図しない上書きを防げる。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **CSS Grid は2次元レイアウトに最適** | `grid-template-columns: repeat(N, size)` で N 列のグリッドを作れる。Flexbox は1次元向き |
| **CSS 変数は JS から動的に設定できる** | `--map-cols` をグリッドの列数に使い、React の `style` 属性から値を流し込む。CSS を書き換えずにマップサイズを変更可能 |
| **Container Query は親要素のサイズで分岐できる** | `@media` は画面全体の幅だが、`@container` は特定の要素の幅。サイドバーがあっても正しく動く |
| **`color-mix(in oklch, ...)` で自然な色混合ができる** | oklch は人間の知覚に基づいた色空間。sRGB より中間色が自然に見える |
| **`@layer` でスタイルの優先度を宣言できる** | base < tiles < player の順にすれば、プレイヤーのスタイルがタイルより常に優先される。書く順序に依存しない |
| **`position: relative` + `absolute` で要素を重ねられる** | タイルに relative を付けて、その上にプレイヤーを absolute で配置する。レイヤー構造の基本パターン |

### 判断のポイント

- **なぜ Canvas ではなく CSS Grid？** → このプロジェクトの目的は「フロントエンド技術の学習」。  
  Canvas は独自の描画 API で HTML/CSS の学習にならない。  
  CSS Grid を使えば、Grid Layout の実践例として学べる。

- **タイルの色をなぜ `color-mix()` で作る？** → 1つの基本色からバリエーションを派生させることで、  
  色の一貫性が保たれ、調整も基本色を変えるだけで済む。  
  ハードコードで個別に色を指定するより保守しやすい。

- **なぜ @layer を使う？** → スタイルが増えてきたとき「どのスタイルが勝つか」が  
  ファイルの書き順に依存すると壊れやすい。  
  @layer で宣言すれば、ファイルを分割・並べ替えしても優先度が変わらない。

```
src/features/map/MapView.module.css を作成:
  - @layer map-base: グリッド構造、Container Query
  - @layer map-tiles: 各タイルの色・質感
  - @layer map-player: プレイヤーの丸い表示
  - レイヤー外: 操作説明のスタイル
```

## Step 4: コンポーネントの入出力を決める — props の設計

### 考えたこと

> MapView は「マップデータ」と「プレイヤーの位置」を受け取って、描画するだけにしたい。  
> キー入力のハンドリングや状態管理は MapView の仕事ではない。  
> → props は `map: MapData` と `playerPos: Position` の2つで十分。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **React は「データは上から下に流れる」（単方向データフロー）** | 親（App）がデータを持ち、子（MapView）は props で受け取って描画する。子が自分でデータを取得しに行かない |
| **プレゼンテーショナルコンポーネントは props だけに依存する** | MapView は「渡されたものを描画する」だけの純粋な表示コンポーネント。テストが容易で再利用性も高い |
| **interface でコンポーネントの API を型定義できる** | `MapViewProps` を定義すれば、使う側が何を渡すべきか明確になる。IDE の補完も効く |

### 判断のポイント

- **map と playerPos をなぜ別の props にする？** →  
  `map` は不変のデータ、`playerPos` はキー入力で頻繁に変わる値。  
  性質が異なるものを分けておくと、将来 `React.memo` で「map が変わっていなければ再計算しない」といった最適化がしやすくなる。

- **MapView に移動ロジックを持たせないのはなぜ？** →  
  「表示」と「ロジック」を分離するため。  
  MapView をミニマップ表示に再利用するとき、移動ロジックは不要。  
  テスト時もキーボードイベントを模擬する必要がなく、座標を渡すだけで確認できる。

```
MapViewProps を設計:
  - map: MapData       ← 描画するマップ
  - playerPos: Position ← プレイヤーの現在座標
```

## Step 5: コンポーネントの中身を実装する — JSX + CSS

### 考えたこと

> Grid のセル数は JS のデータで決まるから、CSS 変数で渡そう。  
> タイルの2次元配列は flatMap で1次元にして、Grid に流し込めばいい。  
> プレイヤーの表示は、該当タイルの子要素として absolute 配置の div を出す。  
> アクセシビリティのために role と aria-label もつけておこう。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **React の `style` 属性で CSS 変数を設定できる** | `style={{ '--map-cols': map.width }}` で JS の値を CSS に渡す。className で静的なスタイル、style で動的な値、という使い分け |
| **`flatMap()` で2次元配列を平坦化できる** | CSS Grid は子要素を左上から順に自動配置する。2次元配列を保つ必要がないので flatMap で1次元にする |
| **React のリストレンダリングには `key` が必要** | key があると React は「どの要素が変わったか」を O(n) で特定できる。なければ全要素を比較し直す |
| **条件付きレンダリングは三項演算子で書ける** | `{isPlayer ? <div .../> : null}` で、該当タイルにだけプレイヤーを表示 |
| **`Record<TileType, string>` で全キーの網羅を強制できる** | タイル種別→CSSクラスの対応表。新しいタイルを追加したとき、対応を書き忘れると型エラーになる |
| **`role` と `aria-label` でアクセシビリティを確保できる** | `<div>` はセマンティクスがないため、role="grid" / role="gridcell" でグリッド構造を明示する |

### 判断のポイント

- **プレイヤーをなぜ別の `<div>` で重ねる？** →  
  タイルの背景色をそのまま残しつつ、プレイヤーを上に載せたいから。  
  タイル自体のスタイルを変える方法だと、プレイヤーが去ったときに元の色に戻す処理が必要。  
  別要素なら、React が勝手に追加・削除してくれる。

- **CSS クラスの対応表をなぜコンポーネントの外に置く？** →  
  レンダリングのたびにオブジェクトを生成するのは無駄。  
  定数はモジュールスコープに置いて、一度だけ作る。

```
src/features/map/MapView.tsx を作成:
  - TILE_CLASS: タイル種別→CSSクラスの対応表
  - MapView: JSX で Grid を構築し、条件付きでプレイヤーを描画
```

## Step 6: 状態管理とユーザー入力をつなげる — カスタムフック

### 考えたこと

> プレイヤーの座標と向きが状態。キー入力に応じて変化する。  
> position と direction は同時に変わるので、useState より useReducer が適切。  
> キーボードイベントの監視は useEffect で window に addEventListener する。  
> このロジックをカスタムフックに切り出せば、App がすっきりする。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **`useReducer` は「複数の値が関連して変化する」場合に適する** | position と direction は移動操作で同時に変わる。useState を2つ使うと更新が分散して管理しにくい |
| **reducer は純粋関数** | `(state, action) → newState` の形。副作用（API呼び出し等）を含まない。テストしやすい |
| **`useEffect` で DOM イベントリスナーを管理できる** | マウント時に addEventListener、アンマウント時に removeEventListener。クリーンアップ関数でメモリリークを防ぐ |
| **`useCallback` で関数の参照を安定化できる** | handleKeyDown が毎レンダリングで新しく作られると、useEffect が毎回再実行される。useCallback で防ぐ |
| **`e.code` は IME に影響されない** | `e.key` は IME オン時に `'Process'` になる。`e.code` は物理キー位置を返すので、日本語入力中でも WASD が効く |
| **React の状態はイミュータブルに更新する** | `state.position.x = 5` はNG。`{ ...state, position: { x: 5, y: state.position.y } }` のように新しいオブジェクトを返す |
| **カスタムフックでロジックを分離できる** | コンポーネントに直接書くと肥大化する。フックに切り出せば再利用・テストが容易 |

### 判断のポイント

- **なぜ useState ではなく useReducer？** →  
  `useState` は1つの値を管理するのに適している。  
  ここでは「移動」という1つのアクションで position と direction の両方が変わる。  
  reducer に集約すれば「移動操作の全ロジック」が1つの switch 文に収まる。  
  将来 `'attack'` や `'interact'` アクションを追加するのも自然。

- **reducer をなぜクロージャにする？** →  
  壁判定にマップデータが必要。useReducer の reducer は本来外部の状態に依存しない純粋関数。  
  `createReducer(map)` でマップデータをクロージャに閉じ込めることで、純粋関数の性質を保ちつつマップを参照できる。

- **e.preventDefault() をなぜ direction が有効な時だけ呼ぶ？** →  
  矢印キーのスクロールは防ぎたいが、Tab キーのフォーカス移動や Escape キーなどのブラウザデフォルト動作は妨げたくない。  
  「ゲーム操作に使うキーだけ」デフォルト動作を無効化する。

```
src/features/map/usePlayerMovement.ts を作成:
  - DELTA: 方向→座標変化量の定数
  - CODE_TO_DIRECTION: キーコード→方向の Map
  - usePlayerMovement(): useReducer + useEffect + useCallback を組み合わせたフック
```

## Step 7: 全部を組み合わせる — App コンポーネント

### 考えたこと

> 部品が全部揃った。App で組み合わせるだけ。  
> マップデータを読み込み、カスタムフックで状態を管理し、MapView に渡す。  
> App は「何を組み合わせるか」だけを記述する薄いレイヤーにしたい。

### 使った前提知識

| 知識 | どう使ったか |
|------|-------------|
| **ルートコンポーネントは薄くする** | App はデータの取得と子コンポーネントへの受け渡しだけ。描画の詳細は MapView、ロジックは usePlayerMovement が担当 |
| **`<main>` タグでメインコンテンツを示す** | `<div>` には意味がない。`<main>` はスクリーンリーダーやSEOに「ここが主コンテンツ」と伝える |

```
src/App.tsx:
  - grasslandMap をインポート
  - usePlayerMovement(grasslandMap) で状態取得
  - <MapView map={grasslandMap} playerPos={player.position} /> で描画
```

## まとめ: 思考の骨格

```
┌─────────────────────────────────────────────────────┐
│  1. 登場人物を型で定義する                            │
│     「何があるか」を先に決めると、以降の全コードで      │
│     型チェックが効く                                  │
├─────────────────────────────────────────────────────┤
│  2. データを定義する                                  │
│     型に従ってデータを書く。型があるのでミスは          │
│     コンパイラが検出してくれる                          │
├─────────────────────────────────────────────────────┤
│  3. 見た目を設計する（CSS）                           │
│     2次元 → Grid、重ね合わせ → position、             │
│     動的な値 → CSS変数、応答性 → Container Query       │
├─────────────────────────────────────────────────────┤
│  4. コンポーネントの API を決める（props）             │
│     「何を受け取り、何を表示するか」を interface で宣言  │
│     表示だけに徹する（プレゼンテーショナル）             │
├─────────────────────────────────────────────────────┤
│  5. コンポーネントを実装する（JSX）                    │
│     データ → JSX要素への変換。flatMap, map, 条件分岐    │
├─────────────────────────────────────────────────────┤
│  6. ロジックをフックに切り出す                         │
│     useReducer で状態、useEffect で副作用、            │
│     useCallback で安定化。全部フックに閉じ込める        │
├─────────────────────────────────────────────────────┤
│  7. App で結合する                                    │
│     データ + フック + コンポーネント を繋ぐだけの        │
│     薄いレイヤー                                      │
└─────────────────────────────────────────────────────┘
```

この「型 → データ → CSS → props → JSX → フック → 結合」の流れは、  
React + TypeScript で新しい機能を作るときの汎用的な思考パターンとして使える。
